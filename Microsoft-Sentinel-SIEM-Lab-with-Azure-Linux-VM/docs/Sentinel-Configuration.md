# Microsoft Sentinel and AMA Configuration Guide

This guide covers how to enable Microsoft Sentinel, install the Azure Monitor Agent, and configure Data Collection Rules (DCR) to collect syslog events.

---

## 🛡️ Step 1: Enable Microsoft Sentinel

Microsoft Sentinel runs on top of a Log Analytics Workspace. Enabling it unlocks SIEM capabilities, threat detection rules, workbooks, and playbooks.

1. In the search bar of the **Azure Portal**, type `Microsoft Sentinel`.
2. Click **Create** or **Add**.
3. Select your workspace `log-shubh` from the list.
4. Click **Add** to enable Sentinel.

---

## 🔌 Step 2: Install Azure Monitor Agent (AMA)

The Azure Monitor Agent (AMA) replaces legacy agents and runs as an extension on the virtual machine. It gathers system log events and sends them to Azure Monitor.

### Method A: Via Azure Portal (Recommended)
1. Go to your virtual machine resource page (`vm-shubh`).
2. On the left navigation pane under **Settings**, click on **Extensions + Applications**.
3. Click the **+ Add** button at the top.
4. Search for **Azure Monitor Linux Agent**.
5. Select it, click **Next**, and click **Create** (using default configuration).
6. Wait for the deployment to finish until the status shows **Succeeded**.

### Method B: Via Azure CLI
Alternatively, deploy the extension using the CLI:
```powershell
az vm extension set \
  --resource-group rg-shubh \
  --vm-name vm-shubh \
  --name AzureMonitorLinuxAgent \
  --publisher Microsoft.Azure.Monitor \
  --version 1.0 # Or latest version
```

---

## ⚙️ Step 3: Configure Data Collection Rule (DCR)

A Data Collection Rule (DCR) specifies which events should be collected and where they should be sent.

### Setup Data Connector
1. Open **Microsoft Sentinel** and select your workspace `log-shubh`.
2. In the navigation menu, under **Configuration**, click **Data Connectors**.
3. Search for **Syslog via AMA**.
4. Click on the connector row and select **Open connector page**.
5. Under the Configuration tab, click **+ Create data collection rule**.

### Configure DCR Settings
1. **Basics**:
   - **Rule Name**: `syslog-dcr-shubh`
   - **Subscription**: Your subscription
   - **Resource Group**: `rg-shubh`
2. **Resources**:
   - Click **+ Add resource(s)**.
   - Expand your subscription and select your virtual machine `vm-shubh`.
   - Click **Apply**.
3. **Collect**:
   - Set the minimum log levels for each of the syslog facilities to capture the necessary logs.

## 📋 Syslog Facilities Reference

Syslog categorizes messages by the source of the message (Facility) and its severity level (Severity). 

### Configured Facilities
| Facility | Description | Use Case |
|---|---|---|
| `LOG_AUTH` | Authorization and authentication systems (e.g., login programs). | Tracking user logins, su attempts, etc. |
| `LOG_AUTHPRIV` | Secure or private authorization messages. | Capturing sensitive access changes, SSH key authorizations. |
| `LOG_DAEMON` | System daemon processes without explicit facilities. | Monitoring background service errors and starts. |
| `LOG_KERN` | Kernel messages. | Tracking kernel-level errors, hardware events. |
| `LOG_SYSLOG` | Internal messages generated by the syslog service itself. | Monitoring logging subsystem performance. |
| `LOG_USER` | Generic user-level messages. | Capturing events generated by user applications (e.g., `logger`). |

### Log Levels / Severities
By selecting `LOG_DEBUG` as the minimum log level for these facilities, we capture **all** logs from debug messages up to emergencies. In a production environment, you should tune these levels (e.g., to `LOG_WARNING` or `LOG_ERR`) to control data ingestion costs, except for critical security tables like `LOG_AUTH` and `LOG_AUTHPRIV` where security events are logged at information levels.
